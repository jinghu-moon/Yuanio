# Claude/Codex/Gemini 对话式 UI 实施方案 (v3)

> 在 Yuanio Android 端实现高性能的对话式 Agent UI：Thinking/Reasoning 折叠块、Step Group 视觉增强、Token 计数、Markdown 渲染，并严格对齐 Claude Code / Codex / Gemini CLI 的事件流模型。

**v3 变更摘要**：在 v2 基础上新增 4 项关键修正与 3 项来源对齐：
1. `toolUseId` 改为 **tool_call + tool_result 双向携带**，消除状态关联断链。
2. `seenToolUseIds.clear()` 改为 **仅在 turn 结束时清理**（非 `tool_result`）。
3. Thinking/Step/Token 设计从“仅 Claude 对齐”升级为 **Claude/Codex/Gemini 统一事件语义**。
4. 引入 **高频流式渲染节流策略**（16ms 基线 + backlog catch-up），参考 Codex TUI chunking 机制。

### 1.5 源码取证结论（Claude/Codex/Gemini）

> 取证来源：`refer/agent-source-code/*` 中的实际实现与协议定义，不做猜测。

| Agent | 关键输出机制 | 对 Yuanio 的直接约束 |
|------|-------------|----------------------|
| Claude Code | `stream-json` + `--include-partial-messages`，`assistant`/`result(tool_result)` 交错；`tool_use.id` 与 `tool_result.tool_use_id` 成对出现 | 必须做 `tool_use` 去重，且必须保留 `toolUseId` 全链路，不能在每个 `result` 都 reset |

| Gemini CLI | `useGeminiStream` 中把 `Thought`、`Content`、`ToolCallRequest`、`ToolResult` 分开处理；并通过 `findLastSafeSplitPoint` 控制大段流式文本切分 | 需要支持 Thinking 独立轨道 + 大文本切片，避免长消息导致整屏重组闪烁 |

## 5. Phase 2：CLI 适配器升级

### 5.1 claude-adapter.ts — thinking 提取 + tool_use 去重

**文件**: `packages/cli/src/adapters/claude-adapter.ts`

完整重构，关键新增：thinking 提取、tool_use id 去重、turnId 传递。

```typescript
import type { AdapterFn, NormalizedEvent } from "./types";

let lastText = "";
let hadStreamedText = false;
let lastThinking = "";                // 新增：thinking 去重
const seenToolUseIds = new Set<string>(); // 新增：tool_use 去重（修复 Issue #3）

export function resetClaudeState() {
  lastText = "";
  hadStreamedText = false;
  lastThinking = "";
  seenToolUseIds.clear();
}

export const claudeAdapter: AdapterFn = (raw) => {
  const msg = raw as Record<string, any>;
  if (!msg || typeof msg !== "object") return [{ kind: "raw", data: raw }];

  const events: NormalizedEvent[] = [];

  if (msg.type === "assistant") {
    const content: any[] = msg.message?.content ?? [];
    const turnId: string | undefined = msg.message?.id;  // assistant 消息 ID（修复 Issue #4）

    // 1. 提取 thinking blocks（新增）
    //    partial message 模式下 thinking 是累积替换，与 text 相同逻辑
    const thinkings = content.filter((b) => b.type === "thinking");
    const fullThinking = thinkings.map((b) => b.thinking ?? "").join("");
    if (fullThinking && fullThinking !== lastThinking) {
      events.push({ kind: "thinking", thinking: fullThinking, turnId });
      lastThinking = fullThinking;
    }

    // 2. 提取 text blocks（已有，增量去重）
    const texts = content.filter((b) => b.type === "text");
    const fullText = texts.map((b) => b.text).join("");
    if (fullText && fullText !== lastText) {
      const delta = fullText.startsWith(lastText)
        ? fullText.slice(lastText.length)
        : fullText;
      if (delta) {
        events.push({ kind: "text", text: delta, partial: true });
        hadStreamedText = true;
      }
      lastText = fullText;
    }

    // 3. 提取 tool_use blocks — 按 id 去重（修复 Issue #3）
    //    --include-partial-messages 下每次重发完整 content[]，
    //    同一个 tool_use（相同 id）会在多次 assistant 消息中重复出现。
    //    仅在首次见到时发送 "running" 事件。
    const tools = content.filter((b) => b.type === "tool_use");
    for (const t of tools) {
      const toolUseId: string | undefined = t.id;
      if (toolUseId && seenToolUseIds.has(toolUseId)) continue; // 跳过已发送的
      if (toolUseId) seenToolUseIds.add(toolUseId);
      events.push({
        kind: "tool_call",
        tool: t.name ?? "unknown",
        params: { ...t.input ?? {} },
        status: "running",
        toolUseId,
      });
    }

  } else if (msg.type === "result") {
    const isToolResult = msg.subtype === "tool_result";
    if (isToolResult) {
      const result = typeof msg.content === "string"
        ? msg.content
        : JSON.stringify(msg.content);
      events.push({
        kind: "tool_result",
        tool: msg.tool_name ?? "unknown",
        result,
        status: "done",
        toolUseId: msg.tool_use_id,
      });
    }

    // 仅在 turn 结束时 reset，不能在 tool_result 时 reset（修复 dedupe 清理时机）
    if (!isToolResult) {
      const wasStreamed = hadStreamedText;
      lastText = "";
      hadStreamedText = false;
      lastThinking = "";
      seenToolUseIds.clear();

      // 提取 usage 信息（已有）
      if (msg.usage) {
        events.push({
          kind: "usage",
          inputTokens: msg.usage.input_tokens ?? 0,
          outputTokens: msg.usage.output_tokens ?? 0,
          cacheCreationTokens: msg.usage.cache_creation_input_tokens,
          cacheReadTokens: msg.usage.cache_read_input_tokens,
        });
      }

      // 仅在没有流式传输过文本时才发送 result（避免重复）
      if (msg.result && !wasStreamed) {
        events.push({ kind: "text", text: String(msg.result) });
      }
    }

  } else if (msg.type === "system" && msg.subtype === "hook") {
    events.push({
      kind: "hook_event",
      hook: msg.hook ?? "unknown",
      event: msg.message ?? "",
      tool: msg.tool,
    });
  } else {
    events.push({ kind: "raw", data: raw });
  }

  return events;
};
```

**关键变更：**
- **thinking 提取**：与 text 相同的增量去重逻辑（`lastThinking` 比较）
- **tool_use id 去重**：`seenToolUseIds` Set 确保同一 tool_use 只发送一次 `running` 事件（**修复 Issue #3**）
- **turnId 传递**：`msg.message.id` 作为 thinking 合并 key 传递到 Android 端（**修复 Issue #4**）
- **reset 时机修正**：仅在 **turn-end result** 重置；`subtype=tool_result` 只更新工具状态，不清理去重集合
- **tool_result 关联**：`tool_result.tool_use_id` 透传到 `NormalizedEvent.tool_result.toolUseId`

```kotlin
            items.indexOfLast { it is ChatItem.Thinking }

            items[existingIdx] = (items[existingIdx] as ChatItem.Thinking)
                .copy(content = thinking)
        } else {
```

```kotlin
"tool_call" -> {
    val obj = JSONObject(payload)
    val itemAgent = obj.optString("agent").takeIf { it.isNotBlank() } ?: _agentState.value.agent
    val status = obj.getString("status")
    val toolUseId = obj.optString("toolUseId").takeIf { it.isNotBlank() }

    if (viewingActive) {
        if (status == "done" || status == "error") {
            // tool_result → 更新已有 ToolCall 的状态和结果
            val result = obj.opt("result")?.takeIf { it != JSONObject.NULL }?.toString()
            val items = _items.value.toMutableList()
            val idx = if (toolUseId != null) {
                // 优先按 toolUseId 精确匹配
                items.indexOfLast { it is ChatItem.ToolCall && it.toolUseId == toolUseId }
            } else {
                // 降级：匹配最后一个 running 的同名工具
                val toolName = obj.getString("tool")
                items.indexOfLast { it is ChatItem.ToolCall && it.status == "running" && it.tool == toolName }
            }
            if (idx >= 0) {
                val existing = items[idx] as ChatItem.ToolCall
                items[idx] = existing.copy(status = status, result = result)
                _items.value = items
            } else {
                // 兜底：找不到 running 项时也要落一条结果，避免状态丢失
                val toolName = obj.getString("tool")
                addItem(ChatItem.ToolCall(
                    tool = toolName,
                    status = status,
                    result = result,
                    summary = ToolSummary.formatOneLiner(toolName, null),
                    toolUseId = toolUseId,
                    agent = itemAgent
                ))
            }
        } else {
            // running → 添加新 ToolCall 项
            val toolName = obj.getString("tool")
            val params = obj.optJSONObject("params")
            addItem(ChatItem.ToolCall(
                tool = toolName,
                status = status,
                result = null,
                summary = ToolSummary.formatOneLiner(toolName, params),
                toolUseId = toolUseId,
                agent = itemAgent
            ))
        }
    }
}
```

```kotlin
"usage_report" -> {

        val usage = obj.optJSONObject("usage")

            val taskId = obj.optString("taskId").takeIf { it.isNotBlank() }

            val items = _items.value.toMutableList()
            val existingIdx = items.indexOfLast {
                it is ChatItem.UsageInfo && it.taskId == taskId
            }
            val info = ChatItem.UsageInfo(
                inputTokens = usage.optInt("inputTokens", 0),
                outputTokens = usage.optInt("outputTokens", 0),
                cacheCreationTokens = usage.optInt("cacheCreationTokens", 0),
                cacheReadTokens = usage.optInt("cacheReadTokens", 0),
                taskId = taskId,
                agent = _agentState.value.agent
            )
            if (existingIdx >= 0) {
                items[existingIdx] = info
                _items.value = items
            } else {
                _items.value = items + info
            }
        }
    }
}
```

## 8. Phase 5：端到端集成与边缘场景

### 8.1 Thinking 合并策略详解（修复 Issue #4）

**问题重述**：v1 方案的 "替换 items.lastOrNull() as? Thinking" 在以下场景失败：

```
时间线 →
[Thinking "分析中..."]              ← items[-1] = Thinking ✓
[Thinking "分析中..."] [ToolCall]   ← items[-1] = ToolCall ✗ 合并失败！
```

**v3 解决方案**：

1. **Claude adapter** 提取 `msg.message.id` 作为 `turnId`
2. **Android 端** 使用 `indexOfLast { it is Thinking && it.turnId == turnId }` 精确定位

**边缘场景处理**：
- **无 turnId**（旧版 Claude Code / Codex / Gemini）：回退到 `indexOfLast { it is Thinking }` — 最后一个 Thinking，与 v1 行为一致
- **turnId 变化**（新一轮对话）：`indexOfLast` 找不到 → 创建新 Thinking 项
- **清理时机**：`resetClaudeState()` 在 **turn-end result** 调用；`tool_result` 期间不清理去重状态

### 8.2 Tool Result 关联（不变）

`dispatch.ts:44-53` 中 `tool_result` 通过 `TOOL_CALL` 信封发送（status="done"），Android 端通过 `toolUseId` 匹配已有的 `running` ToolCall 并更新状态。参见 §6.2.2。

### 8.3 消息顺序保证

一轮 assistant 回复的事件顺序（Claude Code stream-json）：

```
thinking (多次 partial)  → Thinking 块（按 turnId 合并）
tool_use (running)       → ToolCall 项（按 id 去重）
tool_result (done)       → 更新 ToolCall 状态
text (partial, 多次)     → STREAM_CHUNK → 流式文本
usage                    → USAGE_REPORT → UsageInfo 项（累计覆盖）
result                   → STREAM_END → 重置状态
```

渲染顺序（自然保证）：
```
[Thinking]               ← 先到
[ToolCallGroup: N steps] ← tool_use 紧随
[MessageBubble: AI 文本]  ← text 最后到
[TokenCountBadge]         ← usage_report 收尾
```

### 8.4 高频流式渲染策略（新增，参考 Codex TUI chunking）

目标：在高频 chunk（例如 80ms 甚至更快）下，避免 Compose 列表因每 token 重组而抖动。

策略：

1. **提交节拍**：默认每 `~16ms`（约 60fps）合并一次待渲染文本增量，不做逐事件立即提交。
2. **Catch-up 模式**：当待处理队列深度或最老事件等待时间超过阈值（例如 depth>=8 或 age>=120ms），一次提交多条增量，快速追平。
3. **退出滞回**：队列回落后保持 200~300ms 再退出 catch-up，防止模式抖动。
4. **顺序不变**：仅改变“提交批次”，不改变事件顺序。

实现位置建议：
- `ChatViewModel` 内新增 stream commit coordinator（不侵入 UI 组件）
- `ChatMessageList` 保持纯渲染，避免承担节流逻辑

### 8.5 历史持久化策略（修复 Issue #5）

**决策：Thinking / UsageInfo 为会话临时数据，不持久化。**

**现状分析**：
- `persistHistory()` 仅序列化 `ChatItem.Text` → `List<Pair<String, String>>`
- `ChatHistory.save()` 存储 `[{t, c}, ...]` JSON 数组
- 改造 ChatHistory 存储模型成本高（需兼容旧数据、迁移策略、回滚测试）

**方案**：
- `persistHistory()` **保持现状**：仅持久化 Text 项
- Thinking / ToolCall / UsageInfo 在**重新打开聊天时不恢复**（与当前 ToolCall / FileDiff / Approval 行为一致）
- 用户重新连接后，只看到对话文本历史，这与 claude-remote 的行为相同（刷新页面后仅保留文本）

**未来可选优化**（不在本期范围）：
- 扩展 ChatHistory 为类型化存储：`[{_type: "text", ...}, {_type: "thinking", ...}]`
- 需要数据迁移和版本兼容，作为独立 Phase 规划

```
    └─ handleEnvelope: thinking / tool_call 增强 / usage_report
    │
    ▼
Phase 4: Android UI 组件                                  ← 渲染层
    ├─ ThinkingBlock.kt: 可折叠 thinking
    ├─ ToolCallGroupView: 视觉增强（图标 + shimmer + StepItem）
    ├─ TokenCountBadge.kt: token 展示
    └─ ChatMessageList.kt: 新类型渲染分支
    │
    ▼
Phase 5: 集成验证
    └─ 端到端测试 + 边缘场景验证
```

Phase 1-2 为 CLI 侧，Phase 3-5 为 Android 侧，可以由不同角色并行开发（只要 shared types 先定义好）。

## 12. 设计决策记录

| 编号 | 决策 | 理由 | 备选方案 |
|------|------|------|---------|
| D1 | 基于现有 `ToolCallGroupView` 增强 | 避免重复建设，ChatMessageList.kt 已有 `buildToolCallGroups` + `DisplayGroup.ToolCallGroup` | 新建 StepGroupCard.kt → 与现有代码重复 |
| D2 | 复用 `USAGE_REPORT` 而非新增 `ASSISTANT_META` | dispatch.ts 已发送累计 token（`UsageReportPayload`），Android 端仅需接收 | 新增 ASSISTANT_META → 协议重复（Issue #2） |
| D3 | `seenToolUseIds` Set 去重 tool_use，且仅 turn-end 清理 | partial message 模式下 content[] 每次包含所有已发出的 tool_use，`tool_result` 期间仍可能重复回放 | 每个 result 都清理 → 产生重复 ToolCall（Issue #3） |
| D4 | turnId 基于 `msg.message.id` | 同一轮 assistant 消息共享 id，跨中间项也能定位 | "替换最后一项" → 中间插入 ToolCall 后合并失败（Issue #4） |
| D5 | Thinking/UsageInfo 不持久化 | ChatHistory 仅存 `Pair<String,String>`，改造成本高且 claude-remote 也不持久化 | 扩展存储模型 → 需数据迁移+版本兼容（Issue #5） |
| D6 | StepItem 使用 `step.summary`（预计算） | summary 在 addItem 时由 `ToolSummary.formatOneLiner(toolName, params)` 生成，params 已可用 | 在 StepItem 中调 `ToolSummary.generate(tool, null)` → 丢失 params 信息（Issue #6） |
| D7 | `toolUseId` 放入 `ToolCallPayload` 而非 `params._toolUseId` | 规范字段比 hack 污染 params 更清晰；向后兼容（undefined 即可） | `params._toolUseId` hack → 下游需特殊解析 |
| D8 | 单个 ToolCall 保留 `DisplayGroup.Single` | 与现有 `buildToolCallGroups` 阈值一致（≥2 分组） | 所有 ToolCall 都分组 → 改动范围增大 |
| D9 | 引入 16ms 提交节拍 + catch-up 滞回 | 高频流式输出下减少 Compose 重组抖动，兼顾低延迟和稳定性 | 逐事件即时提交 → 大段输出时明显闪烁 |

## 附录 A：Claude Code stream-json 事件格式参考

一次完整交互的事件流示例（含 message.id）：

```jsonl
{"type":"system","subtype":"init","version":"2.1.39","session_id":"abc123"}
{"type":"assistant","message":{"id":"msg_01A","content":[{"type":"thinking","thinking":"让我分析..."}],"model":"claude-sonnet-4-20250514"}}
{"type":"assistant","message":{"id":"msg_01A","content":[{"type":"thinking","thinking":"让我分析这段代码..."},{"type":"tool_use","id":"tu_1","name":"Read","input":{"file_path":"src/utils.ts"}}]}}
{"type":"result","subtype":"tool_result","tool_name":"Read","content":"file contents...","tool_use_id":"tu_1"}
{"type":"assistant","message":{"id":"msg_02B","content":[{"type":"thinking","thinking":"文件内容表明..."},{"type":"tool_use","id":"tu_1","name":"Read","input":{"file_path":"src/utils.ts"}},{"type":"tool_use","id":"tu_2","name":"Edit","input":{"file_path":"src/utils.ts","old_string":"...","new_string":"..."}}]}}
{"type":"result","subtype":"tool_result","tool_name":"Edit","content":"ok","tool_use_id":"tu_2"}
{"type":"assistant","message":{"id":"msg_03C","content":[{"type":"text","text":"我已经完成了优化..."}],"usage":{"input_tokens":1200,"output_tokens":350},"stop_reason":"end_turn"}}
{"type":"result","result":"我已经完成了优化...","usage":{"input_tokens":1200,"output_tokens":350}}
```

**关键观察**：
- `msg.message.id` 在同一轮的 partial 更新中保持不变（如 `msg_01A` 出现两次）
- 新一轮对话产生新 id（`msg_02B`, `msg_03C`）
- `tool_use.id`（如 `tu_1`）在第二条 `msg_02B` 中重复出现 → 需要 `seenToolUseIds` 去重

## 附录 B：与 claude-remote 的架构差异

| 维度 | claude-remote | Yuanio |
|------|-------------|---------|
| 桥接方式 | JSONL transcript 文件监听 | 子进程 stdout stream-json |
| 通信协议 | 原始 WebSocket | Socket.IO + E2E 加密信封 |
| 权限审批 | HTTP Hook (hold-open) | 信封 APPROVAL_REQ/RESP |
| 前端技术 | HTML/CSS/JS (Tauri WebView) | Kotlin Jetpack Compose |
| 事件格式 | Anthropic Messages API 原始 | NormalizedEvent 标准化 |
| 缓存策略 | IndexedDB DOM 快照 | EncryptedSharedPreferences JSON |
| 多 Agent | 仅 Claude | Claude + Codex + Gemini |
| tool_use 去重 | transcript 天然去重（文件追加） | adapter 侧 `seenToolUseIds` Set |
| Thinking 合并 | DOM 直接操作（单个 div 更新） | turnId + indexOfLast 定位 |

Yuanio 的架构优势：
1. **多 Agent 统一接口**：Codex 和 Gemini 也能享受 StepGroup UI
2. **E2E 加密**：消息全程加密，relay 无法读取
3. **原生 Compose**：性能和可访问性优于 WebView
4. **离线队列**：断线自动补发，不丢消息
