## 🔴 严重级别：高风险问题 (Critical)

### C1. 生产环境密钥管理缺陷

**问题描述**:
端到端测试需要 `JWT_SECRET` 环境变量，但项目中缺乏密钥管理的标准化流程和文档。

**风险评估**:
- **安全风险**: 🔴 高
- **影响范围**: 认证系统、生产部署
- **潜在后果**:
  - 密钥泄露可导致身份伪造
  - 密钥丢失导致服务不可用
  - 密钥轮换困难

**证据**:
```
验证报告 10.1 节：
"baseline:protocol:e2e 需要 JWT_SECRET 环境变量"
"生产环境需要独立配置密钥"
```

**修复建议**:
1. 创建 `.env.example` 模板，明确标注必需的环境变量
2. 添加密钥生成脚本 `scripts/generate-secrets.sh`
3. 在 README.md 中添加"生产部署安全检查清单"章节
4. 实现密钥轮换机制（建议使用 Vault 或 AWS Secrets Manager）

**修复优先级**: 🔴 立即处理（部署前必须完成）

---

### C2. 硬编码令牌防护仅在钩子层

**问题描述**:
虽然钩子系统能阻止硬编码令牌写入，但这是最后一道防线。缺乏代码层面的静态分析和 CI/CD 集成。

**风险评估**:
- **安全风险**: 🔴 高
- **影响范围**: 全代码库
- **潜在后果**:
  - 钩子被禁用时失去保护
  - 历史提交中可能已存在泄露
  - 第三方贡献者可能绕过本地钩子

**证据**:
```
验证报告 7.2 节：
"✅ 阻止硬编码令牌"
但仅在 pretool_write 钩子中实现
```

**修复建议**:
1. 集成 `truffleHog` 或 `gitleaks` 进行历史扫描
2. 添加 pre-commit hook（Git 层面）
3. 在 CI/CD 中添加密钥扫描步骤
4. 使用 `eslint-plugin-no-secrets` 进行静态分析

**修复优先级**: 🔴 立即处理

---

### C3. 数据库文件暴露风险

**问题描述**:
项目根目录存在 `yuanio.db`、`yuanio.db-shm`、`yuanio.db-wal` 数据库文件，可能包含敏感数据。

**风险评估**:
- **安全风险**: 🔴 高
- **影响范围**: 用户数据、隐私合规
- **潜在后果**:
  - 数据库文件可能被误提交到版本控制
  - 本地开发环境数据泄露
  - 违反 GDPR/数据保护法规

**修复建议**:
1. 立即检查 `.gitignore` 是否包含 `*.db*` 规则
2. 将数据库文件移至 `data/` 目录
3. 添加数据库加密（SQLCipher）
4. 实现自动备份和清理机制

**修复优先级**: 🔴 立即处理
